What is Planning?

~3 min
Select Simple (3 robots), click Solve & Load, then use Step (⏭) to advance one frame at a time. Focus on Robot 0's path from start → Pickup → Delivery.

Planning Core Concepts

  • State Space — The grid is the state space; each cell is a state
  • Actions — Move up / down / left / right, each with cost = 1
  • Goal — First reach Pickup (P), then reach Delivery (D)
  • A* Search — Uses Manhattan Distance as heuristic to find the shortest path
  • Plan = Action Sequence — The path IS the plan: an ordered sequence of actions
"Planning answers the question: WHAT actions should I take, and in WHAT ORDER?"
2

Why Do We Need Scheduling?

~3 min
Click Play (▶) to run all 3 robots simultaneously. Point out the Conflicts Resolved counter in the stats bar.

Scheduling The Problem

  • If each robot plans independently (ignoring others), their paths will collide
  • Resource Contention — Each grid cell can only hold one robot per timestep
  • Temporal Reasoning — We must consider not just where but when
  • Makespan — Total time until all tasks are done (shown in Time Step)
"Scheduling answers the question: WHEN and WHERE should each agent execute its plan?"
3

Space-Time A* — Bridging Planning & Scheduling

~3 min

Planning + Scheduling The Key Idea

Extend A* from 2D space to 3D space-time, unifying path planning and temporal coordination:

// Standard A* Node = (x, y) // 2D: position only Neighbors = up, down, left, right // Space-Time A* Node = (x, y, t) // 3D: position + time Neighbors = up, down, left, right, WAIT Constraint = "Robot i cannot be at (x,y) at time t"
  • The WAIT action is crucial — a robot can stay in place to let others pass
  • Constraints allow the algorithm to avoid other robots' planned positions
  • Planning (path search) and Scheduling (time coordination) become one unified search

CBS vs. Prioritized Planning

~3 min
Run Medium (6 robots) with CBS, note the Time Step. Then switch to Prioritized Planning and run again. Compare the results — CBS should have fewer total frames and lower total cost.

CBS (Conflict-Based Search)

Optimal
  • 1. Plan each robot independently
  • 2. Detect conflicts between paths
  • 3. Branch: add constraint to agent A or agent B
  • 4. Re-plan affected agent, repeat
  • 5. Return first conflict-free solution

✓ Minimizes total path cost

✗ Exponential worst-case time

Prioritized Planning

Fast but Suboptimal
  • 1. Assign priority order to robots
  • 2. Plan Robot 0 (no constraints)
  • 3. Plan Robot 1 (avoid Robot 0)
  • 4. Plan Robot 2 (avoid 0 & 1)
  • 5. Continue for all robots...

✓ Fast, polynomial time

✗ Later robots get worse paths

Bottleneck — The Scheduling Challenge

~3 min
Switch to Bottleneck (narrow corridor). Point out the 2-cell-wide corridor in the center — the ONLY path between left and right. Click Play and watch robots coordinate through the corridor.

Scheduling Conflict Types

  • Vertex Conflict — Two robots at the same cell at the same time
  • Edge / Swap Conflict — Two robots moving in opposite directions through the same edge (passing through each other)
  • Deadlock — Robots block each other with no way forward

The corridor forces robots to wait, detour, or take turns — this is resource scheduling at its core.

"The corridor is a shared resource, and timing is everything. This is where Scheduling becomes critical."
6

Scaling Up

~2 min
Switch to Large (10 robots, 15 tasks). Click Solve & Load, then Play at 2-3x speed. Let the visual impact speak for itself.

Key Takeaways at Scale

  • Task Allocation (Planning) — Greedy assignment: each task goes to the nearest idle robot
  • Multi-Agent Coordination (Scheduling) — 10 robots navigating without collisions
  • Scalability — CBS becomes expensive with many agents; Prioritized stays fast
  • Real-World Connection — This is exactly what Amazon Robotics / Kiva Systems does
